# 元件命名 Skill 使用指南

## 簡介

這個 skill 協助團隊根據字根字首組合規範來命名 APP 系統元件,確保命名的一致性和可維護性。

## 快速開始

### 1. 準備字根字首表

確保你有最新版本的字根字首表檔案(TSV 格式):
- 檔案名稱: `PAGEs_字典_-_字根字首表.tsv`
- 格式: Tab-separated values (TSV)
- 包含欄位: 中文、對應的英文、分類

### 2. 啟動 Skill

在對話中提及你要命名元件,並上傳字根字首表:

```
我需要為新元件命名,這是我們的字根字首表。
[上傳 TSV 檔案]
```

### 3. 描述元件需求

清楚描述元件的:
- 容器類型(卡片、列表、頁面等)
- 主要功能(顯示、編輯、篩選等)  
- 業務領域(個股、策略、用戶等)
- 特殊屬性(浮動、可折疊等)

### 4. 獲得命名建議

Skill 會提供:
- 符合規範的命名建議
- 詳細的命名解析
- 規則驗證結果
- 當字根不足時的擴充建議

## 使用場景

### 場景 1: 為新元件命名

```
我需要一個顯示個股即時報價的卡片元件,請幫我命名。
```

**輸出**:
```
建議命名: Card_IndividualStock_Quote

命名解析:
- Card (前綴): 卡片容器
- IndividualStock (大類別): 個股
- Quote (子類別): 報價

符合規則: ✓
```

### 場景 2: 檢查現有命名

```
我們有個元件叫 Filter_Container_PersonalSelection,這個命名正確嗎?
```

**輸出**:
```
結果: ❌ 不符合規範

問題: 功能(Filter)出現在前綴(Container)之前,違反順序規則
順序應為: 前綴 → 大類別 → 子類別 → 功能 → 詳細描述

建議修正: Container_PersonalSelection_Filter
```

### 場景 3: 批量命名

```
請幫我命名這些元件:
1. 顯示大盤走勢的圖表
2. 編輯用戶暱稱的輸入框
3. 顯示期貨報價的列表
```

**輸出**: 會提供所有元件的命名建議表格

### 場景 4: 字根不足時

```
需要一個顯示投資組合績效的頁面
```

**輸出**:
```
⚠️ 字根不足

缺少概念: "績效" 在字根表中找不到對應字根

建議新增字根:
- 分類: 子類別
- 中文: 績效
- 英文: Performance
- 用途: 描述投資績效、帳戶表現相關數據

待人工審核後加入字根字首表
```

## 命名規則速查

### 分類順序(由左至右)
```
[前綴] → [大類別] → [子類別] → [功能] → [詳細描述]
```

### 唯一性規則
- 每個分類層級最多出現一次
- 同一層級可以有多個字根(如兩個大類別)

### 常見前綴
- Container: 容器類
- NavBar: 導覽列
- Card: 卡片
- Table: 表格
- Row: 列
- Button: 按鈕
- List: 列表

### 常見功能
- Filter: 篩選
- Edit: 編輯
- Search: 搜尋
- Delete: 刪除
- Collapse: 折疊
- Add: 新增

## 維護字根字首表

### 新增字根流程

1. Skill 識別字根不足並提供建議
2. 團隊討論並確認新字根
3. 將新字根加入 TSV 檔案:
   ```
   編號\t中文\t對應的英文\t分類
   ```
4. 更新後的 TSV 供全團隊使用

### 字根命名原則

- **明確性**: 含義清晰,不易混淆
- **通用性**: 可在多個場景重用
- **一致性**: 風格與現有字根一致
- **簡潔性**: 使用常見、簡短的英文
- **語義性**: 從中文就能理解用途

## 常見問題

**Q: 一定要按照順序嗎?**
A: 是的,順序是硬性規定。前綴必須在最前面,功能必須在大類別之後。

**Q: 可以跳過某些層級嗎?**
A: 可以。例如 `Card_IndividualStock` 就沒有子類別和功能,這是允許的。

**Q: 同一個功能可以出現兩次嗎?**
A: 不可以。同一分類層級不能重複。如果有兩個功能,應該選擇主要的,或拆分成兩個元件。

**Q: 大類別可以出現兩次嗎?**
A: 可以。同一層級的字根可以出現多次,例如 `Card_IndividualStock_News_Collapse`。

**Q: 如何決定哪個字根更重要?**
A: 考慮元件的核心用途。例如「可篩選的個股列表」中,「列表」是結構(前綴),「個股」是內容(大類別),「篩選」是功能。

## 進階使用

### 自訂驗證規則

如果團隊有額外的命名規範,可以在 SKILL.md 中擴充驗證邏輯。

### 整合到開發流程

建議將字根字首表:
1. 放在團隊共享的 Google Drive 或 Notion
2. 設定版本控制
3. 定期 review 和清理不常用的字根
4. 在設計評審時使用此 skill 驗證命名

### 與元件規格 Skill 整合

未來可以創建「元件規格 Skill」,與命名 skill 配合:
1. 先用命名 skill 確定元件名稱
2. 再用規格 skill 產出完整的元件規格文檔
