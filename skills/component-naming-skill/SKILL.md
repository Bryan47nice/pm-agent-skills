# 元件命名 Skill

這個 skill 協助團隊依照字根字首組合規範來命名 APP 系統元件,確保命名的一致性和可維護性。

## 核心功能

1. **載入字根字首表** - 讀取團隊維護的字根字首字典
2. **解析命名需求** - 理解使用者描述的元件功能
3. **建議元件名稱** - 根據規則組合出符合規範的命名
4. **驗證命名正確性** - 檢查是否符合順序和唯一性規則
5. **擴充字根建議** - 當現有字根不足時提供新字根建議

## 字根字首命名規則

### 分類層級與順序

元件名稱由以下五個層級組成,**必須依照此順序**:

```
[前綴] + [大類別] + [子類別] + [功能] + [詳細描述]
```

**順序規則**:
- 從左到右依序為:前綴 → 大類別 → 子類別 → 功能 → 詳細描述
- 不可跳序或倒序(例如:功能不能出現在大類別之前)
- 各層級用底線 `_` 連接

**唯一性規則**:
- 同一個元件名稱中,每個分類層級最多只能出現一次
- 不可出現兩個「前綴」或兩個「功能」

### 命名範例

✅ **正確範例**:
- `Container_Tab` (前綴 + 子類別)
- `NavBar_PersonalSelection` (前綴 + 大類別)
- `Table_Sector_PriceChange` (前綴 + 大類別 + 子類別)
- `Row_ImageTitle_Redirect` (前綴 + 大類別 + 大類別) - **注意**:兩個大類別可以,因為是同一層級
- `Button_Floating` (前綴 + 子類別)
- `Card_Info_Collapse` (前綴 + 大類別 + 功能)

❌ **錯誤範例**:
- `PersonalSelection_NavBar` ❌ (大類別在前綴之前,順序錯誤)
- `Filter_Container_Tab` ❌ (功能在前綴之前,順序錯誤)
- `Container_Pager_Container` ❌ (前綴出現兩次,違反唯一性)
- `Edit_Search_Row` ❌ (功能出現兩次,違反唯一性)

### 字根字首表格式

字根字首表為 TSV 格式,包含以下欄位:
- **中文**: 字根的中文描述
- **對應的英文**: 字根的英文名稱(用於組成元件名稱)
- **分類**: 前綴/大類別/子類別/功能/詳細描述

## 使用流程

### Step 1: 載入字根字首表

在開始命名之前,**必須先要求使用者提供字根字首表檔案**。

```
請提供字根字首表檔案(TSV 格式),讓我載入現有的字根字首定義。
```

載入後,解析並建立字根字首索引:
- 按分類(前綴/大類別/子類別/功能/詳細描述)分組
- 建立中英文雙向對照
- 記錄每個字根的用途說明

### Step 2: 理解元件需求

透過詢問或分析使用者的描述,了解:

1. **元件的容器/結構類型**: 是容器、列表、按鈕、卡片...?
2. **元件的主要功能領域**: 是關於個股、策略、用戶、圖片...?
3. **元件的具體行為**: 是用來篩選、編輯、顯示、切換...?
4. **元件的特殊屬性**: 是浮動的、可折疊的、多選的...?
5. **元件的業務範疇**: 是針對特定指數、特定模組...?

### Step 3: 匹配字根字首

根據需求描述,從字根字首表中尋找匹配項:

```
需求: "一個可以顯示個股相關新聞的卡片,可以折疊"

分析:
- 前綴: Card (卡片)
- 大類別: IndividualStock (個股) + News (新聞)
- 功能: Collapse (折疊)

建議命名: Card_IndividualStock_News_Collapse
```

**匹配策略**:
1. 優先匹配高層級(前綴、大類別)
2. 考慮業務語義,選擇最貼切的字根
3. 避免冗余,能用一個字根表達就不用兩個

### Step 4: 組合與驗證

組合選定的字根並驗證:

**驗證項目**:
- [✓] 順序正確: 前綴 → 大類別 → 子類別 → 功能 → 詳細描述
- [✓] 唯一性: 每個分類層級最多出現一次
- [✓] 語義合理: 名稱能清楚表達元件用途
- [✓] 避免歧義: 不會與現有元件混淆

**輸出格式**:
```
建議命名: Card_IndividualStock_News_Collapse

命名解析:
- Card (前綴): 卡片容器
- IndividualStock (大類別): 個股相關
- News (大類別): 新聞內容
- Collapse (功能): 可折疊

符合規則: ✓
```

### Step 5: 處理字根不足情況

當現有字根無法表達需求時:

```
⚠️ 字根不足提示:

需求: "顯示使用者的投資組合績效曲線圖"
問題: 找不到「績效」相關的字根

建議新增字根:
分類: 子類別
中文: 績效
英文建議: Performance
用途說明: 用於描述投資績效、帳戶表現相關的數據

待人工審核後加入字根字首表。
```

**新增字根的建議原則**:
1. **明確性**: 字根的含義應該清晰且不易混淆
2. **通用性**: 應該能在多個場景重用
3. **一致性**: 命名風格應與現有字根保持一致
4. **簡潔性**: 優先使用常見、簡短的英文單詞
5. **語義性**: 從中文描述就能理解其用途

**提供的資訊包括**:
- 建議的英文名稱
- 應歸屬的分類層級
- 用途說明和使用場景
- 類似的現有字根參考

## 進階功能

### 批量命名

當需要為多個元件命名時:

```
請提供元件清單:
1. 元件描述 A
2. 元件描述 B
3. 元件描述 C

我將一次性提供所有建議命名。
```

### 命名檢查

驗證已有的元件名稱是否符合規範:

```
檢查元件名: Filter_Container_PersonalSelection

結果: ❌ 不符合規範
問題: 功能(Filter)出現在前綴(Container)之前,違反順序規則
建議: Container_PersonalSelection_Filter
```

### 反向查詢

根據元件名稱解析其組成:

```
元件名: Table_Sector_PriceChange_Filter

解析:
- Table (前綴): 表格
- Sector (大類別): 類股
- PriceChange (子類別): 漲跌
- Filter (功能): 篩選

用途推測: 用於篩選類股漲跌數據的表格元件
```

## 字根字首表更新

當新增字根後,提供更新後的字根字首表:

```
新增字根已建議,請將以下內容加入字根字首表的相應分類:

分類: 子類別
編號: [下一個編號]
中文: 績效
對應的英文: Performance
```

## 注意事項

1. **載入字根字首表是必要步驟**: 在開始命名前,必須先載入字根字首表,否則無法提供準確建議

2. **保持客觀中立**: 不要擅自新增字根到字根字首表,只提供建議並由人工審核

3. **考慮業務語境**: 命名應符合團隊的業務領域用語習慣

4. **避免過度組合**: 不是所有層級都必須填滿,保持命名簡潔

5. **說明清楚**: 當提供建議時,要清楚解釋為什麼選擇這些字根

6. **處理歧義**: 當一個需求可能有多種命名方式時,提供多個選項並說明差異

7. **順序優先**: 順序規則是硬性規定,絕對不能違反

8. **同層可重複**: 同一分類層級的字根可以出現多次(如兩個大類別),但不同層級間要保持唯一性

## 常見問題處理

### Q: 如何判斷一個字根屬於哪個分類?

查看字根字首表中的「分類」欄位。如果不確定,可以:
1. 參考類似用途的現有元件
2. 詢問使用者該字根的主要用途
3. 根據語義判斷(容器通常是前綴,行為通常是功能)

### Q: 什麼情況下需要建議新增字根?

當使用者的需求中包含:
1. 現有字根無法精確表達的概念
2. 新的業務領域或功能模組
3. 需要更細緻的分類顆粒度

### Q: 如何處理縮寫和全稱?

優先使用字根字首表中已定義的形式。如果表中有縮寫(如 VIP),就使用縮寫;如果是全稱,就使用全稱。保持一致性。

### Q: 元件名稱可以有多長?

理論上沒有長度限制,但實務上建議:
- 3-4 個部分為佳(如 `Container_Tab_Filter`)
- 最多不超過 5-6 個部分
- 避免過度冗長,保持可讀性

## 使用範例

### 範例 1: 基本命名

```
使用者: 我需要一個展示個股報價資訊的卡片

分析:
- 容器類型: Card (卡片) → 前綴
- 功能領域: IndividualStock (個股) → 大類別
- 資料類型: Quote (報價) → 子類別

建議命名: Card_IndividualStock_Quote

命名解析:
- Card (前綴): 卡片容器
- IndividualStock (大類別): 個股
- Quote (子類別): 報價

符合規則: ✓
```

### 範例 2: 複雜命名

```
使用者: 需要一個可以篩選並編輯自選股列表的導覽列元件

分析:
- 容器類型: NavBar (導覽列) → 前綴
- 功能領域: PersonalSelection (自選股) → 大類別
- 操作行為: Filter (篩選) + Edit (編輯) → 功能

問題: 兩個功能行為,需要選擇主要的或拆分成兩個元件

方案 A: NavBar_PersonalSelection_Filter (主要功能為篩選)
方案 B: NavBar_PersonalSelection_Edit (主要功能為編輯)
方案 C: 拆分為兩個元件:
  - NavBar_PersonalSelection_Filter
  - NavBar_PersonalSelection_Edit

建議: 如果篩選和編輯是並重的功能,建議拆分為兩個元件
```

### 範例 3: 字根不足情況

```
使用者: 需要一個顯示使用者持倉分析報告的頁面

分析:
- 容器類型: Page (頁面) → 前綴
- 相關概念: 
  - Acount (用戶) → 大類別 ✓
  - "持倉" → 找不到對應字根 ❌
  - "分析報告" → 找不到對應字根 ❌

⚠️ 字根不足,需要擴充

建議新增字根:

| 分類 | 中文 | 英文建議 | 用途說明 |
|------|------|---------|---------|
| 子類別 | 持倉 | Position | 使用者的股票持有部位、庫存狀態 |
| 子類別 | 分析報告 | AnalysisReport | 數據分析結果的報告呈現 |

建議命名(待字根加入後): Page_Acount_Position_AnalysisReport

或簡化為: Page_Acount_Position (如果分析報告是預設行為)
```

## 總結

這個 skill 的核心價值在於:
1. **標準化**: 確保團隊使用一致的命名規範
2. **效率化**: 快速產生符合規範的元件名稱
3. **可擴充**: 支援字根字首表的持續擴充
4. **可驗證**: 提供命名正確性檢查機制

透過系統化的命名流程,讓團隊成員能夠快速、準確地為新元件命名,降低溝通成本,提升開發效率。
